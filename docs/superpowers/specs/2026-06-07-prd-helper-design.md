# PRD Helper —— 设计文档 (Design Spec)

- **日期**:2026-06-07
- **作者**:CarterJia
- **状态**:已评审,待实现
- **一句话**:把用户脑中模糊的想法,通过协作式对话,变成一份结构化、诚实、可决策的 PRD 大纲。

---

## 1. 背景与目标

用户做产品工作时,常需要把脑中模糊的想法落成结构化的 PRD。本工具让用户用自然语言描述需求,自动生成结构化 PRD 大纲。

题目把四件事留给我们决策,本设计逐一作答:

1. PRD 大纲包含哪些模块、结构如何定义 → 见 §4
2. 用户描述模糊、信息不足怎么办 → 见 §3 与 §7
3. 交互流程:单轮还是多轮 → 见 §3(选"协作者"混合模型)
4. 输出格式与排版 → Markdown 为唯一事实源,见 §6

**定位**:面向 AI 产品 / AI PM 场景。核心差异化不在排版,而在**对 LLM 能力边界的判断**——懂得处理信息缺口、绝不编造事实。

---

## 2. 设计原则:什么是"一份好的 PRD 大纲"

本工具的每个功能都对应下列某一条原则。这也是产品说明的主线。

1. **结构完整**——覆盖一份专业 PRD 应有的模块(背景、用户、目标、需求、边界……)。
2. **对不确定性诚实**——明确区分"事实"与"假设":能推断的标注为假设供用户确认;属于客观事实而工具无从得知的(如负责人姓名),留占位符,**绝不编造**。这是 AI 产品的底线。
3. **决策导向**——好 PRD 由"包含什么"和"排除什么"共同定义。强制包含**非目标 (Non-Goals)** 与功能**优先级 (P0/P1/P2)**。
4. **足够具体可执行**——避免正确的废话,功能拆到可被研发/设计接手的颗粒度。
5. **是能迭代的活文档**——通过多轮对话逐步精修,而非一次性产出。

---

## 3. 交互模型:协作者(混合)

核心洞察:**"自然语言 → PRD"真正的难点不是排版,而是一份好 PRD 需要的信息,用户脑子里往往还没想清楚。** 偷懒的工具会把一句模糊的话"扩写"成看似专业、实则全是幻觉假设的文档。

因此采用**协作者模型**,把"处理模糊输入"做成产品亮点:

1. 用户输入一段自然语言描述。
2. 工具**立即生成一版 PRD 草稿**(即时见价值)。
3. 草稿中**显式标注所有假设**;同时在对话区给出 **3–5 个最关键的澄清问题**。
4. 用户回答问题或直接补充 → 工具**重新生成**,假设收敛、内容更准。
5. 循环精修,直到用户满意。

对比与取舍:

- 纯单轮(输入即生成):最简单,但面对模糊输入要么沉默假设、要么产出空洞。
- 纯多轮(先追问再生成):质量高,但用户在看到任何产出前就被连环追问,容易流失。
- **协作者(本方案)**:先给草稿再精修,兼顾即时反馈与逐步收敛,且把题目最难的两个考点(模糊输入 + 交互设计)直接变成核心卖点。

## 4. PRD 大纲结构定义

分三类:**自动生成(元信息)**、**核心模块(始终生成)**、**按需模块(LLM 判断相关才生成)**。

### 4.0 文档信息 (Document Metadata) — 自动 / 占位

| 字段 | 生成规则 |
|------|----------|
| 负责人 / 作者 | **占位符 `[待填写]`**——属客观事实,不编造 |
| 评审 / 研发 / 设计负责人 | **占位符 `[待填写]`** |
| 创建时间 / 最后修改时间 | 自动填系统时间;每次重新生成刷新"最后修改时间" |
| 版本号 | 默认 `V0.1`,每次重新生成自增(V0.1 → V0.2) |
| 状态 | 默认 `草稿` |

> 这一节体现核心原则 §2.2:**该填的填、该留白的留白、绝不编造人名。**

### 4.1–4.8 核心模块(始终生成)

1. **TL;DR 一句话概述**——做什么、为谁、解决什么。
2. **背景与问题**——为什么现在做、痛点是什么。
3. **目标用户与场景**——典型用户、使用场景。
4. **目标与成功指标**——可量化的成功标准 (KPI)。
5. **非目标 (Non-Goals)**——本版明确不做什么。(原则 §2.3)
6. **功能需求**——拆成具体功能,每条带**优先级 (P0/P1/P2)** 与用户故事。
7. **用户流程**——关键路径(文字描述,可选 mermaid)。
8. **待澄清问题 (Open Questions)**——工具识别出的信息缺口,与 §3 的追问一一对应。

### 4.9 按需模块(相关才生成)

- **非功能需求**(性能 / 安全 / 合规)——偏 B2B 或技术型产品时。
- **风险与依赖**。
- **里程碑 / 粗排期**。

**假设的呈现**:凡 LLM 基于推断补充的内容,在所属模块内以醒目悬浮块标注:`> 💡 假设:……`。

## 5. 系统架构

### 5.1 技术栈

- **Next.js (App Router) + TypeScript + Tailwind CSS**——单仓库前后端,易部署(Vercel 一键)。
- **Anthropic Claude API**——服务端调用。
- **流式 (streaming)** 生成——文档在画布中逐字渲染。

### 5.2 布局(方案 B:左对话 / 右画布)

```
┌─────────────────────────────────────────────┐
│  PRD Helper            📋 复制 MD   ⬇️ 下载   │
├──────────────────┬──────────────────────────┤
│  💬 对话           │  📄 PRD 文档画布           │
│  - 用户气泡        │  (Markdown 实时渲染)       │

```

### 5.4 生成机制与数据流

1. 前端 `POST /api/generate`,body:`{ description: string, history: Turn[] }`(`history` 含历次问答,用于精修)。
2. 服务端用 `lib/prompt.ts` 组装 system prompt(含 §4 结构定义、§2 原则、假设/追问产出要求)。
3. 调用 Claude **流式**接口,响应体为文本流。
4. **模型输出契约**:先输出完整 **Markdown PRD**(含假设悬浮块与"待澄清问题"节),最后追加一段以固定分隔符包裹的机器可读 JSON:

   ```
   <<<PRD_META>>>
   { "assumptions": ["..."], "questions": ["..."] }
   <<<END_PRD_META>>>
   ```

5. 前端在流式累积过程中按分隔符切分(`lib/parse.ts` 为纯函数,前后端与测试共用):分隔符之前 → 实时渲染进画布;`<<<END_PRD_META>>>` 到达后解析 JSON → 填充对话区的追问 chips 与"我做了哪些假设"摘要。
6. **版本号与时间由应用层掌握,不由模型生成**:应用维护"重新生成次数"得出版本(V0.1、V0.2……),并取当前系统时间,注入文档信息头对应字段——保证确定性。模型只负责内容,不输出版本/时间。
7. 用户回答 → 追加到 `history` → 再次调用 → 文档重生成、版本自增、修改时间刷新。

> 选择「Markdown 文档 + 结构化尾块」而非纯 JSON 的原因:既能让文档自然地逐字流式渲染(体验好、演示惊艳),又能可靠地把假设/问题结构化喂给对话区。

---

## 6. 输出格式

- **Markdown 为唯一事实源**:画布渲染漂亮排版,导出即所见。
- **📋 复制 Markdown** / **⬇️ 下载 .md**。
- 导出 PDF / Word、推送到 Notion/飞书,列入"再给一周"的优化项,**不在 v1**。

---

## 7. 异常与边界处理(直接回答题目"信息不足怎么办")

| 场景 | 处理 |
|------|------|
| 输入极模糊 / 信息严重不足 | 不硬编内容;生成结构骨架 + 把缺口集中到"待澄清问题",并加重追问 |
| 客观事实未知(人名、具体数字) | 留 `[待填写]` 占位符,**绝不编造** |
| 无 API key | 自动进入 **Demo 模式**(顶部"演示模式"提示),用预置示例模拟完整流程 |
| Claude API 报错(限流/网络) | 对话区友好提示 + 重试按钮 |
| 结构化尾块解析失败 | 降级:文档照常显示,仅跳过 chips 提取(问题仍在文档"待澄清问题"节可见)|
| 输入为空 | 前端拦截,提示先描述需求 |

## 9. 本项目自身的非目标 (Non-Goals)

(自我示范原则 §2.3)

- 不做完整项目管理 / 任务跟踪。
- 不生成 UI 视觉稿或原型。
- v1 不做账号体系与多人协作。
- 不做 PRD 的存储/历史版本管理(刷新即新会话)。
- 不替代 PM 的最终判断——工具是协作者,不是决策者。
